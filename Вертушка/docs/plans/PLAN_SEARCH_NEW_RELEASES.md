# План: Новинки на экране Поиска + сворачиваемая история

## Обзор
Перенести витрину «Новинки» с публичного профиля на вкладку Поиск в Mobile-приложении. История поиска становится сворачиваемой секцией. Хедер Поиска перестраивается в набор «секций», чтобы в будущем можно было легко добавлять блоки рекомендаций выше/ниже без переписывания структуры.

Запросы к Discogs не растут: бэкенд кэширует новинки в Redis на 12 часов (`Backend/app/services/discogs.py:1130`), при открытии Поиска вызов `GET /api/profile/new-releases` отдаст кэш.

## 2. Компонент-обёртка `Section`

**Цель:** дать единый контейнер с заголовком и опциональной свёрткой, чтобы хедер Поиска собирался декларативно.

**Файл:** `Mobile/components/Section.tsx`

**API:**
```tsx
interface SectionProps {
  id: string;                   // для сохранения состояния свёрнутости
  title?: string;                // если задан — рисуется хедер с шевроном
  collapsible?: boolean;         // показывать ли шеврон
  defaultCollapsed?: boolean;    // по умолчанию false
  rightAction?: React.ReactNode; // напр. кнопка "Очистить" в истории
  children: React.ReactNode;
}
```

**Поведение:**
- Если `collapsible`, при тапе по хедеру переключаем состояние, анимация через `LayoutAnimation.configureNext(LayoutAnimation.Presets.easeInEaseOut)` (нативно в RN, без зависимостей).
- Если `title` не задан — рендерим только `children` (для блоков типа AutoRail, у которых свой заголовок).
- Состояние свёрнутости хранится в Zustand (см. п.3), читается по `id`.

## Логика обновления витрины (важно держать в голове)

Кэш **ленивый, не по таймеру**:
- Redis-ключ `new_releases:y{год}_p60` живёт 12 часов (`Backend/app/services/discogs.py:1130`).
- После протухания TTL обновление НЕ происходит само — оно запускается **первым запросом после истечения**. Если 36 часов никто не зашёл — данные не обновятся 36 часов.
- При нормальном трафике (хоть один заход в час) реальная частота апдейтов ≈ раз в 12 часов.
- Между обновлениями все viewers видят один и тот же снапшот (отставание до 12 часов от «горячего» на Discogs).

Сортировка: `sort=want&sort_order=desc` от Discogs, фильтр `format=Vinyl&year={current_year}`. Пул 60 релизов, в выдачу до 12.

На Поиске вызываем `get_new_releases` **без user_id** — глобальный пул, без вычитания коллекции смотрящего (на публичном профиле — с вычитанием). Это осознанно: Поиск показывает «что вообще популярно сейчас».

**Если в будущем понадобится гарантированная свежесть** (без зависимости от того, что кто-то «толкнул» кэш) — поставить cron/Celery beat на прогрев каждые N часов. Сейчас не делаем — текущий лаг до 12ч приемлем для витрины «новинок».

## Не делаем в рамках этой задачи

- Блок «Рекомендации» — оставляем место под него, но реализация отдельной задачей.
- Персональные новинки (по жанрам пользователя) — текущий эндпоинт глобальный, этого хватит для первой итерации.
- Pull-to-refresh для принудительного обновления новинок — кэш и так живёт 12 часов, добавим если понадобится.
- Cron-прогрев кэша — см. секцию «Логика обновления витрины» выше.
