# План: Система достижений «Вертушки»

> **Цель.** Превратить «Вертушку» из утилиты-каталогизатора в живой ритуал коллекционера. Достижения должны не «висеть» как иконки, а **формировать поведение**: подталкивать к глубине (а не к количеству), к открытию редкого, к социальным жестам и к возвращению в приложение в неочевидные моменты.

---

## 0. Философия и принципы

### 0.1. Что такое хорошая ачивка в «Вертушке»

Виниловая культура держится на четырёх китах: **редкость, ритуал, личная история, племя**. Достижения должны попадать хотя бы в один — иначе это ярмарка значков.

| Принцип | Что значит | Чего избегаем |
|---|---|---|
| **Глубина > количество** | «100 пластинок» — слабая ачивка. «10 разных стран пресса в коллекции» — сильная. | Чистые счётчики без качественного сигнала. |
| **Открытие, а не задание** | Большинство ачивок — **скрытые** до выполнения, с лёгкой подсказкой. Первый раз пользователь узнаёт постфактум: «Ого, а так можно было». | Чек-листы как в мобильных играх (DAU-механика). |
| **Уважение к коллекции** | Награды описывают _кто ты как коллекционер_, а не _как часто открываешь приложение_. | Streak-логин-ачивки, «открой 7 дней подряд». |
| **Социальный жест** | Подарки, фолловы, публичный профиль — **первоклассные источники** ачивок. Дарить — престижнее, чем получать. | Ачивки за «накрутку» — followers без ограничений на качество. |
| **Анти-grind** | Если ачивку можно сфармить за вечер скриптом — она не нужна. Все массовые ачивки требуют разнообразия (не просто N штук, а N _разных_). | «Добавь 1000 записей» без условий разнообразия. |
| **Лор** | У каждой ачивки — короткая флавор-строка из виниловой культуры. Не «коллекционер», а «Хранитель Сторон Б». | Сухие технические описания. |

### 0.2. Чего система **не делает**

- Не даёт реальной валюты, премиум-фичи или скидки. Награды — **косметика и статус** (рамки, темы, заголовки публичного профиля, флексные карточки шаринга).
- Не превращает приложение в Duolingo: нет таймеров, нет «сгораний», нет ежедневных квестов в стартовом экране.
- Не штрафует за неактивность.
- Не «просит вспомнить про тебя» назойливо. Push о новой ачивке — **только в момент анлока**.

## 1. Каталог достижений (полный)

Достижений в стартовом каталоге — **84**. Разбиты на 11 категорий. Каждое имеет код, видимость, тир, триггер, прогресс-логику, награду и флавор.

> **Условные обозначения тиров:**
> 🟫 Common (≥30% получат) · ⚪ Uncommon (5–30%) · 🟦 Rare (1–5%) · 🟪 Epic (0.1–1%) · ⭐ Legendary (<0.1%)

> **Hidden** = пользователь не видит ачивку до анлока (только намёк в тематической вкладке: «спрятано 3 ачивки про подарки»).

### 2.3. Анлок-анимация

Когда ачивка получена — **поверх экрана появляется ритуал**:

1. Затемнение фона + лёгкий блюр.
2. Из центра — анимация падающей пластинки (вертикально, с лёгким wobble) и приземления на проигрыватель.
3. Игла опускается → короткий vinyl crackle (опциональный звук, по системному mute).
4. Из центра пластинки разворачивается лента с названием ачивки.
5. Внизу — кнопки «Поделиться» / «В зал трофеев» / «×» (закрыть).

Длительность: ~2.2 сек до интерактивной фазы. Если получено сразу несколько (батч-проверка) — показываем последовательно или агрегатом «+3 ачивки!».

### 2.4. Push-уведомления

- Триггер: только в момент анлока, **не чаще** 1 push-нотификации в час (батчинг).
- Текст: _«🏆 Открыта ачивка: «Хранитель Оригиналов»»._
- Управляется существующим флагом `User.notify_app_updates` (или новым `notify_achievements`).

## 3. Техническая архитектура

### 3.1. Модели данных

**Новая модель** `Backend/app/models/achievement.py`:

```python
class UserAchievement(Base):
    __tablename__ = "user_achievements"

    id = Column(UUID, primary_key=True, default=uuid4)
    user_id = Column(UUID, ForeignKey("users.id", ondelete="CASCADE"), nullable=False, index=True)
    code = Column(String(64), nullable=False, index=True)        # 'C2_first_press_x10'
    unlocked_at = Column(DateTime(timezone=True), nullable=False, default=now)
    progress = Column(Integer, default=0, nullable=False)         # текущее значение счётчика
    progress_target = Column(Integer, default=0, nullable=False)  # целевое
    is_unlocked = Column(Boolean, default=False, nullable=False, index=True)
    metadata = Column(JSONB, nullable=True)  # для динамических — например, имя артиста для H2

    __table_args__ = (
        UniqueConstraint("user_id", "code", name="uq_user_achievement"),
        Index("ix_user_achievements_user_unlocked", "user_id", "is_unlocked"),
    )
```

**Зачем `metadata`:** ачивки `H2`/`F11`/`H7`/`H8` динамические — название зависит от того, _какого_ артиста/жанра/лейбла собрал. В `metadata` храним, например, `{"artist_name": "King Crimson", "artist_id": 12345}`. Один пользователь может получить `H2` несколько раз с разными `metadata` — для этого UniqueConstraint на `(user_id, code)` не подходит для динамических. Решение: для динамических ачивок код имеет суффикс — `H2:king-crimson` (slug). Ниже в §3.5 — как это сериализуется.

**Опциональная вспомогательная модель** `AchievementEvent` (для аналитики и реплея — _не критично для MVP_):

```python
class AchievementEvent(Base):
    __tablename__ = "achievement_events"
    id = Column(UUID, primary_key=True, default=uuid4)
    user_id = Column(UUID, ForeignKey("users.id"), nullable=False, index=True)
    event_type = Column(String(64), nullable=False)  # 'collection_item_added', 'gift_completed', ...
    payload = Column(JSONB)
    created_at = Column(DateTime(timezone=True), default=now, index=True)
```

Эта таблица **не обязательна** для MVP — на запуске считаем напрямую по существующим таблицам. Добавлять, если появится нужда в реплеях / истории.

**Эмиссия события** — из API-обработчиков:

```python
# Backend/app/api/collections.py — после создания CollectionItem:
await emit_event(
    user_id=current_user.id,
    event="collection_item_added",
    payload={
        "collection_item_id": item.id,
        "record_id": record.id,
        "record": record,  # может потребоваться evaluator-ам
    },
    db=db,
)
```

`emit_event` — из `services/achievements/evaluator.py`:

```python
async def emit_event(*, user_id, event, payload, db):
    affected_codes = registry.codes_for_trigger(event)
    for code in affected_codes:
        defn = registry.get(code)
        result = await defn.evaluator(user_id, payload, db)
        if result.unlocked:
            await unlock_achievement(user_id, code, db, metadata=result.metadata)
        elif result.progress is not None:
            await update_progress(user_id, code, result.progress, result.target, db)
    # Анлок-уведомление — после коммита транзакции (используем background_tasks).
```

**Задержки:** evaluator-ы должны быть **дешёвыми** в hot-path. Тяжёлые (категория H — сравнение с дискографией Discogs) делаются на `daily_tick`, а не на каждый `collection_item_added`.

### 3.7. Идемпотентность и race conditions

- При `unlock_achievement` — **upsert** с `is_unlocked=true` по `(user_id, code)`. Двойная эмиссия события не приводит к двойному анлоку.
- При `update_progress` — пишем только если новое значение **больше** текущего (защита от старого/late-arriving события).
- Все evaluator-ы — pure (без сайд-эффектов), сайд-эффекты только в `unlock_achievement`/`update_progress`.

### 3.8. Бэкфилл

После выкатки — **разовый скрипт** `Backend/scripts/backfill_achievements.py`:
- Идёт по всем пользователям.
- Для каждого — синтезирует исторические события (по `created_at` в `CollectionItem`, `Follow`, `GiftBooking`, …) в хронологическом порядке.
- Прогоняет через evaluator с `notify=False` (без push-уведомлений).
- В конце: для каждого пользователя — один консолидированный push «🏆 За время в Вертушке ты заработал N ачивок».

## 4. Анти-фарм и баланс

### 4.1. Защиты от фарма

| Угроза | Защита |
|---|---|
| Создал и сразу удалил 1000 пластинок ради ачивки B3 | Считаем `COUNT(DISTINCT record_id)` от записей `created_at` за последние ≥7 дней с фильтром «не удалена за последние сутки». Альтернатива: hard rate-limit на add (≤50 в час). |
| Накачал followers фейковыми аккаунтами | Для K4/K5 учитываем только followers, у которых есть ≥1 пластинка в коллекции (anti-bot). |
| Бронируем подарки сами себе | `GiftBooking.gifter_email != user.email` обязателен. Уже есть в логике. |
| Просмотры профиля от себя | View не инкрементится, если `viewer_user_id == profile_owner_id`. |
| Подбираем точно 33 пластинки для N1 | Это **специально** простая ачивка-easter egg, фарм здесь приветствуется. |

### 4.2. Калибровка тиров

Перед выкаткой — прогон бэкфилла на тестовой базе и измерение, какой % текущей юзер-базы получит каждую ачивку. Корректировка порогов так, чтобы:

- Common: 30–60% базы
- Uncommon: 5–30%
- Rare: 1–5%
- Epic: 0.1–1%
- Legendary: <0.1%

Если ачивка после бэкфилла попадает мимо тира — корректируем порог (но **не описание**: лор остаётся, цифра подвигается).

## 5. Метрики успеха

### 5.1. Северная звезда

**Доля пользователей, у которых получена хотя бы одна не-foundation ачивка через 30 дней после регистрации.** Цель: ≥40%.

### 5.2. Дополнительные метрики

| Метрика | Что измеряет | Цель |
|---|---|---|
| Среднее число анлоков на пользователя в первые 7 дней | Привлекательность стартового опыта | 4–7 |
| % пользователей, открывших экран «Зал трофеев» | Видимость фичи | ≥60% за 14 дней |
| % share-card нажатий из легендарных анлоков | Виральность | ≥15% |
| Корреляция «получил ачивку H/J» ↔ retention 30 дней | Социальные ачивки → удержание | rₛ ≥ 0.3 |
| Время от добавления записи до анлока B1 | Гладкость UX | <30 сек медиана |
| Доля анлоков от daily_tick vs hot-path | Доля «отложенных» ачивок | <20% (большинство — мгновенные) |

### 5.3. Инструментирование

В `AchievementEvent` (или прямо в логи) пишем для каждого анлока: `code`, `user_id`, `unlocked_at`, `time_since_register`, `time_since_first_record`. Это даёт всю аналитику без отдельного слоя.

## 11. Концептуальные easter eggs

Это **отдельный класс ачивок**: они не про прогресс и не про объём, а про **синхронии, шутки, знание подсказок, культурные отсылки и неожиданные совпадения**. Все скрытые. Цель: чтобы пользователь, наткнувшись, восклицал «не может быть!» и шёл рассказывать.

### 11.1. Принципы концептуальных easter eggs

| Принцип | Что значит |
|---|---|
| **Один раз на пользователя** | Легендарные совпадения не должны повторяться. Если пользователь триггернул — навсегда у него. |
| **Без подсказок в каталоге** | В отличие от обычных hidden, эти **не показываются** ни в каком виде до анлока — даже как «❓ скрытая ачивка». Они — чистый сюрприз. |
| **Триггер из мира, а не из меню** | Нельзя «поставить себе цель» концептуальный ЕЕ. Они срабатывают потому, что мир так устроен — твой день рождения, текущая дата, синхрония с другим пользователем. |
| **Лёгкий, не сакральный** | Если про них узнают и начнут специально подгонять — это не сломает их. Они для тех, кто живёт жизнью, а ачивки догоняют сами. |
| **Ровно одна шкатулка с подвохом** | Фирменная мета-ЕЕ `EE_open_the_box` (см. ниже) — про сам факт того, что система ЕЕ существует. |

### 11.4. Тайный совет: «Лестница» easter eggs

Внутри концептуальных EE есть **скрытая лестница** для самых упорных:

```
Пользователь получает:
  EE_open_the_box  ← первый ЕЕ любого типа
  EE_completionist_ee  ← 15 ЕЕ
  EE_grandmaster  ← 30 ЕЕ (⭐ Legendary, скрытая до анлока EE_completionist_ee)
```

При получении `EE_grandmaster` — **уникальный визуал**: вместо стандартной анлок-анимации с пластинкой — анимация виниловой шкатулки, открывающейся и показывающей мини-музей всех собранных EE. Это единственный сценарий с уникальной анимацией в системе (остальные ачивки — общая «падающая пластинка»).

### 12.2. Доп. таблицы

Для синхронных EE (`EE_synchronicity`, `EE_mentor`, `EE_telepathic`) нужны агрегаты «кто что добавлял когда». **Не делать запрос ко всей `CollectionItem` каждый раз** — слишком дорого. Решение:

**Daily-aggregate таблица** `daily_record_additions`:
```python
class DailyRecordAddition(Base):
    __tablename__ = "daily_record_additions"
    date = Column(Date, primary_key=True)
    record_id = Column(UUID, ForeignKey("records.id"), primary_key=True)
    user_ids = Column(ARRAY(UUID), nullable=False)  # массив пользователей, добавивших в этот день
    count = Column(Integer, nullable=False)
```

При `collection_item_added` — апсертим строку (день, record_id), пополняем массив user_ids и count. Тогда EE_synchronicity = есть строка с count ≥ 2 и моим user_id в array, и среди record-ов с `is_first_press=true OR is_limited=true`.

### 12.3. Lunar phase

Для `EE_full_moon` — предвычисленная таблица `lunar_phases` (date, phase) на 100 лет (генерируется один раз скриптом, занимает ~37 KB в памяти). Альтернатива: библиотека `ephem` или `pylunar` с локальным расчётом — но статичная таблица проще и надёжнее.

### 12.4. Список «пионеров жанра»

Для `EE_woodstock` и подобных — ручные YAML-файлы с фиксированными списками:

```
Backend/app/services/achievements/data/
├── woodstock_artists.yaml      # ~30 имён
├── imperative_verbs.yaml       # ~30 английских + ~15 русских
├── melodiya_aliases.yaml       # ['Melodiya', 'Мелодия', 'Aprelevka', ...]
└── lunar_phases_2000_2100.json # предвычисленная таблица
```

Это **намеренно ручные** данные — Discogs не размечает «выступал ли на Вудстоке».

### 12.5. Мониторинг ложных срабатываний

Концептуальные EE — самые опасные с точки зрения багов (легко выдать неправильно). Перед прод-выкаткой:

1. На бэкфилле логируем **все** срабатывания EE отдельно (`AchievementEvent` с `event_type='ee_unlock_dryrun'`).
2. Ручная проверка: 100 случайных EE-анлоков из бэкфилла — действительно ли условие выполнено?
3. Только потом — реальная выдача.

Для каждого нового EE — отдельный property-based тест с 1000 случайных пользовательских состояний.

## 14. Финальные открытые вопросы (дополнение к §7)

7. **Уникальность EE в масштабе платформы** — для `EE_only_one` (один на платформе обладает) — что делать, если второй пользователь добавил такую же пластинку через год? Отзывать? Предложение: **не отзывать**, дать второму человеку отдельную ачивку `EE_only_two_now` с другим лором.
8. **Дата запуска приложения** — для `EE_vertushka_birthday` нужна фиксированная дата релиза в конфиге.
9. **Granularity «синхрония»** — синхронизация по календарному дню или 24-часовому окну? Предложение: календарный день в TZ пользователя.
10. **Доступность концептуальных EE на публичном профиле** — показывать или скрывать? Предложение: показывать только тем, кто сам уже получил ≥1 EE (создаёт «клуб посвящённых»).
11. **Anti-pattern flag** — некоторые EE (`EE_silence`, `EE_clean_house`) поощряют поведения, противоположные основной цели (молчание, удаление). Не нарушает ли это маркетинговый посыл? Предложение: **нет, наоборот** — это делает систему более человечной и поощряет «дыхание» коллекции.
