## Фаза 1 — 503 ошибки и производительность (Discogs rate limiting)

### 1.1 `_get_price_stats` обходит семафор и retry
- **Файл**: `Backend/app/services/discogs.py:810-824`
- **Проблема**: Прямой `client.get()` вместо `self._get()`. Не соблюдает `_semaphore` (5 concurrent), не ретраит 429/503
- **Фикс**: Переписать `_get_price_stats` через `self._get()`, добавить `curr_abbr` в params и token-авторизацию
- **Влияние**: Основная причина 503 — неконтролируемые параллельные запросы к Discogs

### 1.2 Убрать дублирующий retry на мобилке
- **Файл**: `Mobile/lib/api.ts:80-89`
- **Проблема**: Бэкенд уже ретраит 3 раза (`discogs.py:70-84`), мобилка ещё 3 раза. Итого до 9 запросов и ~13 сек зависания
- **Фикс**: Убрать retry-interceptor для 503 в мобильном клиенте. Бэкенд сам справляется
- **Влияние**: Ускорение ошибочных сценариев с ~13 сек до ~6 сек

### 1.3 N+1 в `get_artist_masters` — 20+ запросов на страницу
- **Файл**: `Backend/app/services/discogs.py:768-771`
- **Проблема**: Для каждого мастер-релиза отдельный `_get_master_info()` запрос. Артист с 20 альбомами = 21 запрос к Discogs
- **Фикс**: Использовать `thumb` из основного ответа через `_thumb_to_cover()` вместо отдельных запросов. Определять тип релиза из `format` поля, а не из треклиста
- **Влияние**: Снижение запросов с 21 до 1 на страницу

## Фаза 2 — Функциональные баги (Mobile)

### 2.1 Search: `Promise.all` падает полностью при одной ошибке
- **Файл**: `Mobile/app/(tabs)/search.tsx:218-221`
- **Проблема**: `Promise.all([search(), searchUsers()])` — если поиск юзеров вернёт 503, результаты поиска пластинок тоже не покажутся
- **Фикс**: Заменить на `Promise.allSettled()`, показывать частичные результаты
- **Влияние**: Поиск пластинок работает даже если поиск юзеров упал

### 2.2 `removeFromCollection` — silent failures в папках
- **Файл**: `Mobile/lib/store.ts` (removeFromCollection)
- **Проблема**: `.catch(() => {})` при удалении из папок. Ошибки проглатываются
- **Фикс**: Логировать ошибки, показывать alert при неудаче удаления из папки

### 2.3 `setSortBy` — нет обработки ошибок
- **Файл**: `Mobile/lib/store.ts` (setSortBy)
- **Проблема**: `fetchCollectionItems()` вызывается без `.catch()`, при ошибке состояние сортировки расходится с данными
- **Фикс**: Добавить `.catch()` с откатом `sortBy` к предыдущему значению

## Фаза 3 — Security

### 3.1 Apple Sign In — токен не верифицируется
- **Файл**: `Backend/app/api/auth.py:221`
- **Проблема**: `identity_token` не проверяется через Apple public keys. Любой может войти под чужим Apple ID
- **Фикс**: Реализовать верификацию через `PyJWT` + Apple JWKS endpoint (`https://appleid.apple.com/auth/keys`)
- **Влияние**: Критическая уязвимость — позволяет захват аккаунтов

### 3.2 `/gifts/my-bookings/by-email` — утечка cancel_token
- **Файл**: `Backend/app/api/gifts.py:225-259`
- **Проблема**: По email без авторизации можно получить все бронирования включая `cancel_token`. Любой может отменить чужое бронирование
- **Фикс**: Убрать `cancel_token` из ответа этого endpoint, либо добавить подтверждение по email (OTP)

### 3.3 Бесконечный цикл в генерации username
- **Файл**: `Backend/app/api/auth.py:242-247`
- **Проблема**: `while True` без лимита итераций
- **Фикс**: Добавить `if counter > 100: raise HTTPException(500, "...")` и fallback на UUID-суффикс

## Фаза 4 — Hardening (технический долг)

### 4.1 Логирование ошибок
- **Файлы**: `Backend/app/api/records.py`, `Backend/app/services/discogs.py`
- **Проблема**: `except Exception: pass` везде — невозможно дебажить продакшен
- **Фикс**: Заменить на `logger.exception(...)` во всех `except` блоках

### 4.2 Валидация длины текстовых полей
- **Файлы**: `Backend/app/schemas/`
- **Проблема**: Нет `max_length` на `bio`, `gifter_message`, `notes` и др.
- **Фикс**: Добавить `Field(max_length=...)` в Pydantic-схемы

### 4.3 Escape wildcards в ILIKE-поиске
- **Файл**: `Backend/app/api/users.py`
- **Проблема**: `%` и `_` в поисковом запросе работают как SQL wildcards
- **Фикс**: Экранировать спецсимволы перед подстановкой в ILIKE

### 4.4 N+1 в поиске юзеров
- **Файл**: `Backend/app/api/users.py`
- **Проблема**: 4 отдельных запроса (followers, following, collection, is_following) на каждого юзера в результатах
- **Фикс**: Переписать на JOIN + subquery, одним SQL-запросом
