# Принципы Вертушки

## Данные пользователей: локалка vs прод

**Локальная база и продакшен — полностью изолированные базы данных. Аккаунты и данные между ними не синхронизируются.**

### Где что хранится

| Среда | URL бэкенда | База данных | Где живут данные |
|---|---|---|---|
| Локальная разработка | `http://<local-ip>:8000/api` | Docker volume `postgres_data` | На твоей машине |
| Продакшен | `https://api.vinyl-vertushka.ru/api` | PostgreSQL на сервере `85.198.85.12` | На сервере |

### Правила

- **Реальную коллекцию строить только на проде** — локальная БД может быть потеряна (удаление Docker volume, переустановка Docker)
- **Локальная БД** нужна только для разработки и тестирования новых фич — данные там временные
- **`docker-compose down -v` удаляет volume** вместе со всеми данными — не запускать, если есть ценные данные
- Деплой (`git push` + скрипт) обновляет только код, данные на проде не трогает
- IP локального бэкенда меняется при смене сети — сверять с `ipconfig getifaddr en0` и обновлять в `Mobile/lib/api.ts`

### Итог

Для разработки — регистрируй тестовый аккаунт на локалке. Реальный аккаунт с коллекцией — всегда на проде.

## Discogs API — изображения

**Discogs CDN использует подписанные imgproxy URL. Менять параметры URL нельзя — подпись сломается, картинка не загрузится.**

### Откуда брать обложки

| Эндпоинт Discogs | Доступные поля | Что использовать |
|---|---|---|
| `/database/search` | `cover_image` (~500px) + `thumb` (150px) | `cover_image` as-is |
| `/masters/{id}` | `images[0].uri` | полноразмерная обложка, использовать напрямую |
| `/releases/{id}` | `images[0].uri` | полноразмерная обложка, использовать напрямую |
| `/artists/{id}/releases` | только `thumb` (150px) | **Не использовать для обложек** — пиксельно |

### Правила

- **Никогда не модифицировать URL** (regex замены `/h:150/` → `/h:500/` и т.п. ломают подпись)
- **`thumb` из `/artists/{id}/releases` не использовать как fallback для обложек** — 150px, пиксельно. Лучше `null` чем пиксель
- **Для поисковой выдачи** — `cover_image` из `/database/search` достаточно
- **Для страницы артиста** — список через `/artists/{id}/releases?type=master&role=Main` (точный artist_id, без текстового шума Search API). Обложки: batch через Search API (`/database/search?type=master&artist={name}&per_page=100`), матчинг по `master_id` (exact). Для мастеров, не найденных в поиске — fallback на `_get_master_info` (`/masters/{id}`). Если и он упал — `cover = None`, не fallback на thumb
- **Search API для обложек страницы артиста — безопасно** при матчинге по `master_id`. Текстовый поиск может вернуть чужие релизы, но мы берём только те, чей `id` совпадает с `master_id` из канонического списка `/artists/{id}/releases`. Ложных совпадений нет. Это заменяет N вызовов `/masters/{id}` (≈100с через rate limiter) на 1 вызов search (≈1с)
- На мобилке `cover_image_url || thumb_image_url` — fallback между двумя полями одного объекта, но оба берутся из HD-источника

### Release type — определение типа релиза

| Источник | Формат данных | Надёжность |
|---|---|---|
| Search API `format[]` | Массив `["Vinyl", "LP", "Album"]` | Высокая — явные дескрипторы |
| Artist releases `format` | Строка `"Vinyl, LP, Album"` | Средняя — парсинг подстрок |
| `/masters/{id}` track count | Число треков → single/ep/album | Низкая — EP с 7 треками = ложный «album» |

Приоритет: строка формата > track count. Метод `_guess_release_type(format_str, title)` используется в `get_artist_masters` первым; track count из `_get_master_info` — только как fallback.

**Никогда не определять тип только по track count без проверки format строки** — это главный источник неправильной классификации EP/синглов как альбомов.

### Таймауты enrichment и rate limiter

`_get_master_info` вызывается для N мастеров через `asyncio.gather`. Все N сразу входят в очередь rate limiter (capacity=55, 0.95 req/s). Последний запрос из 100 ждёт ≈47с.

- **Не оборачивать `_get_master_info` в `asyncio.wait_for`** — таймаут срабатывает раньше, чем rate limiter выдаёт токен, что приводит к постоянным ошибкам (cover=None) и отсутствию кэширования → следующий запрос снова падает в тот же цикл
- **Rate limiter timeout для ENRICHMENT/BATCH приоритетов — 120с** (не 30с), иначе для per_page=100 последние ~40 запросов таймаутятся в очереди ещё до HTTP
- Таймауты делегированы: rate limiter (120с) + httpx (30с на HTTP). Этого достаточно
- `TTL_MASTER_INFO = 7 дней` — первый визит медленный (≈50с для 100 мастеров), последующие мгновенные (кэш). Чем дольше TTL, тем реже платим этот "налог"

### Инвалидация кэша

Ключ `get_artist_masters` содержит версию (`v12:...`). При изменении логики построения результата — инкрементировать версию, чтобы стареющие записи Redis не отдавали данные со старой неправильной классификацией.

### Итог

- Список мастеров артиста: `/artists/{id}/releases?type=master&role=Main` — точный, без дублей
- Обложки: `_get_master_info` → `images[0].uri` из `/masters/{id}`; при ошибке — `null`, не thumb
- Тип релиза: `_guess_release_type(item["format"], item["title"])` → fallback track count
- `/masters/{id}` и `/releases/{id}` — только для экранов деталей одного конкретного релиза
